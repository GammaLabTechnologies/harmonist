# Memory

Persistent project state that survives across Cursor sessions. Three files,
one machine-readable schema, one CLI.

## The files

| File | Kind | Role |
|------|------|------|
| `session-handoff.md` | `state` | Rolling snapshot. Most recent entry = authoritative state. |
| `decisions.md` | `decision` | Append-only architectural decisions. Never edit past entries. |
| `patterns.md` | `pattern` | Lessons — what worked, what didn't. |

Free markdown (headers, intro) is allowed anywhere in the files. The
structured records live between `<!-- memory-entry:start -->` and
`<!-- memory-entry:end -->` markers.

## The contract

See `SCHEMA.md` for the full spec. In short, every entry has:

```yaml
id                — unique identifier
correlation_id    — ties entries from the same task together
at                — ISO 8601 timestamp
kind              — state | decision | pattern
status            — in_progress | done | blocked | rejected
author            — orchestrator | subagent | human
summary           — one-line headline
```

The `correlation_id` is **generated by the enforcement hooks**, not by the
LLM. Read it via `memory.py current-id`.

## Writing entries — always through the CLI

```bash
# From project root
python3 .cursor/memory/memory.py append \
  --file session-handoff \
  --kind state \
  --status done \
  --summary "Wired Stripe webhook handler; added integration tests" \
  --tags payments,backend \
  --body-file /tmp/handoff-body.md
```

The CLI fills in `id`, `correlation_id`, `at` automatically and validates
the resulting block before writing. If validation fails the file is
rolled back, nothing lands half-written.

Other useful commands:

```bash
memory.py current-id                         # print active correlation_id
memory.py latest --file session-handoff --kind state --n 3
memory.py list   --file decisions --correlation 1745333456-0
memory.py show   <entry-id>
memory.py validate [--path <memory-dir>] [--strict]
memory.py bump-task    # hooks use this; advances task_seq by 1
```

## Hand-edits

Direct edits to the markdown are allowed — the validator doesn't care how
an entry got there. But every entry MUST still pass the schema or the
stop hook will block completion. When in doubt, use the CLI.

## Privacy

These files will accumulate project-sensitive detail. Guidance:

- Default: **add `.cursor/memory/*.md` to your `.gitignore`**.
- Files ending `*.shared.md` (e.g. `decisions.shared.md`) are a convention
  for entries you intentionally commit. The validator treats them the
  same way. Review them before pushing.
- Never write raw secrets; use `<PLACEHOLDER>` instead.

## Automatic validation

Two layers:

1. **Hook-driven**: the enforcement hooks under `.cursor/hooks/` call
   `validate.py` whenever a memory file changes. Invalid entries block
   the `stop` gate.
2. **CI**: `hooks/tests/` runs `validate.py` against the shipped templates;
   `.gitlab-ci.yml` and `.github/workflows/` run it on every PR.

## Migration

Schema v1 is the first structured version. When a breaking change lands
it will ship alongside `migrate.py` similar to `agents/scripts/migrate_schema.py`.
